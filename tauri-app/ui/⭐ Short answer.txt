⭐ Where qmd fits inside SDOA
Surface Layer
UI never calls qmd directly.
It only triggers workflows like:

“Search project”

“Find related files”

“Explain this module”

“Refactor this component”

The UI stays dumb — workflows do the work.

Domain Layer
Domain objects don’t know qmd exists.

They only know:

“search results”

“snippets”

“file references”

“ranked relevance”

This keeps your domain pure and portable.

Orchestration Layer
This is where qmd becomes a first‑class workflow step.

Examples:

SearchProjectWorkflow

FindRelatedFilesWorkflow

ExplainModuleWorkflow

RefactorWorkflow

IngestFolderWorkflow

Each workflow calls the qmd adapter, not qmd directly.

Access Layer
This is where qmd lives.

You create a module:

Code
/access/qmd/index.js
This module exposes:

Code
search(query)
vsearch(query)
query(query)
get(file)
init(path)
Every workflow uses this adapter — never the CLI directly.

This keeps your architecture clean, portable, and testable.

Step 1: Install qmd

First, install Bun if you don’t have it:

curl -fsSL https://bun.sh/install | bash

Then install qmd:

bun install -g https://github.com/tobi/qmd

Step 2: Index your projects

For each project you work on:

qmd collection add ./my-project — name myproject

This indexes all markdown and JSON files by default. You can customize the file patterns with — mask:

qmd collection add ./my-project — name myproject — mask “**/*.{ts,tsx,md,json}”

Then generate vector embeddings for semantic search:

qmd embed

You can verify everything is indexed:

qmd status
Step 3: Add qmd as an MCP server in Claude Code

Open your Claude Code MCP config at ~/.claude/mcp.json and add qmd:

{
  "mcpServers": {
    "qmd": {
      "command": "qmd",
      "args": [
        "mcp"
      ]
    }
  }
}
